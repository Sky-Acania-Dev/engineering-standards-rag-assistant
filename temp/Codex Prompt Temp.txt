Restart from the current main branch, which already has table formatting but no footnote system. Do NOT attempt a full end-to-end footnote implementation in one pass.

The previous attempts proved that char-based superscript anchor detection is viable, but page-bottom footnote-content detection and integration are unstable. The main regressions were:
- ordinary numbered lists misclassified as footnotes
- tables misclassified as footnotes
- duplicate table text leaking into body chunks
- section/chapter drift caused by false-positive footnote extraction
- missing first footnote body on a page causing id shifts
- two-digit body labels like 10/11/14/15/22/23/24 being split or swallowed into content

Implement this in four phases, and stop after each phase with tests/debug artifacts.

PHASE 1 — CHAR-LEVEL ANCHOR DETECTION ONLY
Add only char-based superscript anchor detection using pdfplumber chars.
Do not link anchors to bodies.
Do not strip text.
Do not emit footnote metadata into chunks.
Do not modify body text.

For each page, output a debug structure with:
- page
- detected anchors
- anchor id
- bbox
- line index
- nearby anchor text
- confidence
- flags:
  - line_final
  - punctuation_adjacent
  - heading_like

Success conditions for phase 1:
- detects title anchor 1 on page 5
- detects anchor 2 after “40 CFR Part 745.”
- detects anchors 10, 11, 14, 24
- does not corrupt chunk text

PHASE 1 is done at this point. All footnote anchors in the pdf are detected. We also noticed that at least one footnote has the anchor and content on different pages.

PHASE 2 — PAGE-BOTTOM REGION CLASSIFICATION ONLY
Do not link anything yet.

Detect candidate page-bottom regions and classify each as one of:
- true_footnote_block
- ordinary_numbered_list
- table_region
- unknown

If classified as true_footnote_block, parse body labels and content into a page-local map.
If not, do not parse as footnotes.

For each page, output debug:
- region bbox
- classification
- reasons for classification
- parsed body labels if true_footnote_block

Hard negative rule:
Do NOT classify a region as footnotes if it is:
- a normal numbered requirements list
- a continued numbered standards list
- a table or table-adjacent row block
- normal section prose near page bottom

Success conditions for phase 2:
- page 7 bottom region classified as true_footnote_block with bodies 2,3,4,5
- page 8 contractor requirements classified as ordinary_numbered_list, not footnotes
- DWH EF table page classified as table_region, not footnotes
- no bogus ids like 80, 200, 48

PHASE 2 is done at this point. All footnote contents at the end of the pages in the pdf are detected with anchor id properly identified.

PHASE 3 — LINKING ONLY
Use only:
- anchors from phase 1
- footnote bodies from phase 2

Link by exact page-local id match only.
Do not shift ids by relative order.
If body 2 is missing, 3 must remain 3, 4 remain 4, etc.

Still do not modify chunk text yet.

Output debug:
- anchor ids
- body ids
- resolved links
- dropped anchors with reason

Success conditions for phase 3:
- page 5 anchor 1 links to body 1
- page 7 2/3/4/5 do not shift
- footnote 6 does not merge into 7

PHASE 4 — INTEGRATION INTO CHUNKS
Only after phases 1–3 pass.

Integrate conservatively:
- strip only validated true footnote blocks from body reconstruction
- insert [footnote: N] only for resolved links
- emit structured footnotes metadata
- do not emit unlinked fallback metadata unless the body came from a validated true_footnote_block
- exclude table regions from body text if already emitted as table chunks
- never duplicate a table both inline and as table content

Preserve:
- chunk ordering
- section/chapter metadata
- table formatting

Hard rules:
- no ordinary numbered list may become footnote metadata
- no table rows may become footnote metadata
- no table text may be duplicated into body text
- no id shifting when one anchor/body is missing
- no corruption like §2306.514 or 21.6[footnote:24]
